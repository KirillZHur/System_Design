## 1. Анализ производительности

В рамках лабораторной работы рассматривается LMS-сервис — система управления обучением. Сервис предоставляет REST API для регистрации и авторизации пользователей, получения пользовательских данных, работы с курсами, уроками, записями на курсы и отметками прогресса.

Основные хранилища данных:
- PostgreSQL — пользователи, авторизация, записи на курсы, прогресс;
- MongoDB — курсы и уроки;
- Redis — кеширование часто читаемых данных и хранение счётчиков rate limiting.

###  Часто выполняемые операции

К часто выполняемым операциям относятся endpoints, которые вызываются при обычной работе пользователя с системой:

| Endpoint | Описание | Причина высокой нагрузки |
|---|---|---|
| `POST /api/v1/auth/login` | Авторизация пользователя | Может вызываться часто, чувствителен к brute-force |
| `GET /api/v1/users/by-login/{login}` | Получение пользователя по логину | Часто используется после авторизации и при проверке профиля |
| `GET /api/v1/users/search` | Поиск пользователей | Может создавать нагрузку на PostgreSQL при поиске по маске |
| `GET /api/v1/courses` | Получение списка курсов | Часто открывается студентами и преподавателями |
| `GET /api/v1/courses/{course_id}/lessons` | Получение уроков курса | Часто вызывается при просмотре курса |
| `GET /api/v1/users/{user_id}/courses` | Получение курсов пользователя | Часто используется в личном кабинете |

### Медленные операции

Потенциально медленными являются операции, связанные с обращением к базам данных:

1. Чтение пользователя из PostgreSQL по логину.
   Несмотря на наличие индекса по логину, при большом количестве запросов постоянные обращения к PostgreSQL создают лишнюю нагрузку.

2. Поиск пользователей по имени и фамилии.
   Поиск по маске может быть медленнее точечного поиска, особенно если используется `LIKE`/`ILIKE`.

3. Получение списка курсов и уроков из MongoDB.
   Эти данные могут часто читаться разными пользователями и при этом изменяться не очень часто.

4. Авторизация пользователя.
   Endpoint `POST /api/v1/auth/login` обращается к PostgreSQL и дополнительно является чувствительным к большому количеству повторных запросов.

### Требования к производительности

Для системы LMS выбраны две ключевые метрики производительности:

| Метрика | Описание | Целевое значение |
|--------|----------|------------------|
| Время отклика (Response time) | Время от отправки запроса до получения полного ответа | ≤ 200 мс для чтения, ≤ 400 мс для записи |
| Пропускная способность (Throughput) | Количество запросов, обрабатываемых системой в секунду | ≥ 100 RPS (чтение), ≥ 50 RPS (запись) |

#### Обоснование

- **Время отклика ≤ 200 мс** — считается комфортным для пользователя (интерактивные системы)  
- Запросы на запись медленнее (из-за БД), поэтому допускается до 400 мс  
- **Пропускная способность** выбрана исходя из учебной нагрузки (десятки–сотни пользователей)

#### Ожидаемый эффект от Redis

Использование Redis позволит:
- снизить время отклика для чтения до ~50–100 мс
- увеличить пропускную способность в 2–5 раз за счёт снижения нагрузки на БД

## 2. Проектирование стратегии кеширования

Для повышения производительности системы используется кеширование с применением Redis.

В системе реализуются два шаблона работы с кешем:

### Сквозное чтение (Read-through)

При выполнении операции чтения (например, получение пользователя по логину) используется следующий алгоритм:

1. Проверка наличия данных в Redis (кеш).
2. Если данные найдены — возвращаются из кеша.
3. Если данных нет:
   - выполняется запрос в PostgreSQL;
   - полученные данные сохраняются в Redis;
   - данные возвращаются клиенту.

Преимущества:
- уменьшение количества обращений к PostgreSQL;
- снижение времени отклика;
- увеличение пропускной способности системы.

### Сквозная запись (Write-through)

При создании или изменении данных (например, регистрация пользователя):

1. Данные сохраняются в PostgreSQL.
2. После успешной записи данные сохраняются в Redis.

Преимущества:
- консистентность данных между кешем и базой данных;
- отсутствие "устаревших" данных в кеше.

### Выбор данных для кеширования

В качестве кандидатов для кеширования выбраны часто запрашиваемые данные:

- пользователь по логину (`GET /api/v1/users/by-login/{login}`);
- список курсов (`GET /api/v1/courses`);
- список уроков курса (`GET /api/v1/courses/{course_id}/lessons`).

Наиболее критичным является endpoint получения пользователя по логину, так как он используется:
- после авторизации;
- при проверке прав доступа;
- при работе с профилем.

### Время жизни кеша (TTL)

Для кешируемых данных устанавливается ограниченное время жизни:

- пользователь: 5–10 минут;
- списки курсов и уроков: 1–5 минут.

Это позволяет:
- уменьшить нагрузку на базы данных;
- избежать хранения устаревших данных;
- обеспечить баланс между актуальностью и производительностью.

## 3. Реализация кеширования

### Реализация кеширования в REST API

В системе реализовано кеширование с использованием Redis в качестве in-memory key-value хранилища.

Кеширование применяется для ускорения операций чтения и снижения нагрузки на PostgreSQL и MongoDB.

Используется шаблон **сквозного чтения (read-through cache)**:

1. При поступлении запроса сервис сначала проверяет наличие данных в Redis.
2. Если данные найдены — они возвращаются из кеша.
3. Если данные отсутствуют:
   - выполняется запрос в базу данных;
   - результат сохраняется в Redis;
   - данные возвращаются клиенту.

Данные сериализуются в JSON и сохраняются в Redis с заданным временем жизни (TTL).

### Применение кеширования для endpoint'ов

Кеширование реализовано для следующих endpoint'ов:

#### 1. Получение пользователя по логину

GET `/api/v1/users/by-login/{login}`


- ключ в Redis:  
  `user:login:{login}`  
- источник данных: PostgreSQL  
- TTL: ~300 секунд  

#### 2. Поиск пользователей

GET `/api/v1/users/search`


- ключ в Redis:  
  `users:search:{query}`  
- источник данных: PostgreSQL  
- TTL: ~60 секунд  

Для обоих endpoint'ов используется read-through стратегия.

### Инвалидация кеша

Для обеспечения актуальности данных реализована инвалидация кеша при изменении данных.

#### При регистрации пользователя:

POST `/api/v1/auth/register`

Выполняются следующие действия:

1. Новый пользователь сохраняется в PostgreSQL.
2. Кеш пользователя обновляется (write-through):
   - создаётся/обновляется ключ `user:login:{login}`.
3. Кеш результатов поиска пользователей инвалидируется:
   - удаляются ключи вида `users:search:*`.

Это позволяет избежать ситуации, когда в кеше находятся устаревшие данные.

### Нагрузочное тестирование

Для оценки производительности системы используется утилита wrk.

Тестируется endpoint:
GET `/api/v1/users/by-login/{login}`

Данный endpoint выбран, так как:
- часто используется;
- обращается к PostgreSQL;
- подходит для кеширования.

#### Подготовка

Перед тестированием:
1. Создан тестовый пользователь.
2. Получен JWT-токен.
3. Сервис запущен без использования кеша Redis.

#### Команда для тестирования

```bash
wrk -t1 -c50 -d30s \
  -H "Authorization: Bearer <TOKEN>" \
  http://localhost:8080/api/v1/users/by-login/user1
```

где:

-t — количество потоков (1, 5, 10)

-c — количество соединений

-d — длительность теста

####  План эксперимента

Проводятся замеры при:

1 потоке
5 потоках
10 потоках

Фиксируются:

- среднее время ответа (Latency)
- Requests per second (RPS)

В качестве метрик производительности выбраны:
- время отклика (Response time);
- пропускная способность (Throughput).

При проведении нагрузочного тестирования с использованием утилиты wrk данные метрики измеряются следующим образом:

- время отклика оценивается через среднее значение Latency;
- пропускная способность оценивается через показатель Requests per second (RPS).

Таким образом:
- Latency ≈ Response time;
- RPS = Throughput.

#### Сравнение производительности (без кеша / с кешем)

| Потоки | Latency без кеша (ms) | Latency с кешем (ms) | RPS без кеша | RPS с кешем |
|----------|----------------------|---------------------|--------------|-------------|
| 1       | 7                    | 8                   | 1412         | 1266        |
| 5       | 17                   | 13                  | 1502         | 1947        |
| 10       | 35                   | 23                  | 1415         | 2154        |

- При малом числое потоков использование кеша не даёт прироста,
  так как PostgreSQL справляется с запросами без значительных задержек.

- При увеличении потоков:
  - время отклика уменьшается:
    - с 35 мс до 23 мс (при 10)
  - пропускная способность возрастает:
    - с ~1400 RPS до ~2150 RPS

- Это связано с тем, что Redis:
  - работает быстрее PostgreSQL;
  - снижает нагрузку на базу данных;
  - уменьшает время обработки запроса.

Таким образом, использование кеширования эффективно при средней и высокой нагрузке.

## 4. Проектирование rate limiting

Rate limiting используется для защиты API от перегрузки и злоупотреблений.  
В LMS-сервисе наиболее критичными являются endpoints, которые могут часто вызываться автоматически или использоваться для перебора данных.

### Endpoints, требующие rate limiting

| Endpoint | Причина ограничения | Уровень риска |
|---|---|---|
| `POST /api/v1/auth/login` | Возможен brute-force подбор пароля | Высокий |
| `POST /api/v1/auth/register` | Возможна массовая регистрация пользователей | Средний |
| `GET /api/v1/users/search` | Может создавать нагрузку на PostgreSQL при поиске по маске | Средний |
| `GET /api/v1/users/by-login/{login}` | Часто вызывается и может использоваться для перебора логинов | Средний |

Основной endpoint для реализации rate limiting в рамках лабораторной работы:

```http
POST /api/v1/auth/login
```

Он выбран, потому что:

- связан с авторизацией;
- обращается к PostgreSQL;
- наиболее чувствителен к большому числу повторных запросов;
- может использоваться для brute-force атаки.

### Выбор алгоритма rate limiting

Для endpoint POST `/api/v1/auth/login` выбран алгоритм Fixed Window Counter.

Принцип работы:

- Время делится на фиксированные окна, например по 60 секунд.
- Для каждого клиента в Redis хранится счётчик запросов.
- При каждом запросе счётчик увеличивается.
- Если количество запросов превышает лимит, сервис возвращает ошибку 429 Too Many Requests.
- После завершения временного окна счётчик автоматически удаляется за счёт TTL.

### Обоснование выбора Fixed Window Counter

Алгоритм Fixed Window Counter выбран по следующим причинам:

- простая реализация;
- хорошо подходит для учебного проекта;
- легко реализуется через Redis INCR + EXPIRE;
- не требует хранения истории всех запросов;
- потребляет мало памяти.

**Недостаток алгоритма** — возможен всплеск запросов на границе временных окон.
Для текущей лабораторной работы этот недостаток допустим, так как цель — реализовать базовую защиту от частых повторных запросов.

### Лимиты запросов

Для LMS-сервиса выбраны следующие лимиты:

| Endpoint                             | Алгоритм             |                    Лимит | Ключ Redis                             |       TTL |
| ------------------------------------ | -------------------- | -----------------------: | -------------------------------------- | --------: |
| `POST /api/v1/auth/login`            | Fixed Window Counter |  10 запросов / 60 секунд | `rate_limit:login:{client_ip}`         | 60 секунд |
| `POST /api/v1/auth/register`         | Fixed Window Counter |   5 запросов / 60 секунд | `rate_limit:register:{client_ip}`      | 60 секунд |
| `GET /api/v1/users/search`           | Fixed Window Counter |  60 запросов / 60 секунд | `rate_limit:search:{client_ip}`        | 60 секунд |
| `GET /api/v1/users/by-login/{login}` | Fixed Window Counter | 120 запросов / 60 секунд | `rate_limit:user_by_login:{client_ip}` | 60 секунд |


В рамках реализации будет ограничен endpoint:
```http
POST /api/v1/auth/login
```

### Поведение при превышении лимита

Если пользователь превышает лимит запросов, сервис должен вернуть:
```http
HTTP/1.1 429 Too Many Requests
```
Также в ответ добавляются HTTP-заголовки:
```http
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 0
X-RateLimit-Reset: <seconds_until_reset>
```
Где:

- X-RateLimit-Limit — максимальное число запросов в окне;
- X-RateLimit-Remaining — сколько запросов осталось;
- X-RateLimit-Reset — через сколько секунд лимит будет сброшен.

### Ожидаемый эффект

Rate limiting позволит:

- защитить endpoint авторизации от brute-force запросов;
- снизить риск перегрузки PostgreSQL;
- ограничить влияние одного клиента на остальных пользователей;
- повысить стабильность API при резких всплесках нагрузки.

## 5. Реализация rate limiting

В рамках данного этапа был реализован механизм ограничения количества запросов (rate limiting) для endpoint:
```http
POST /api/v1/auth/login
```

### Выбранный алгоритм

Для реализации был выбран алгоритм Fixed Window Counter (фиксированное временное окно).

### Хранение состояния

Для хранения счётчиков используется Redis.

Ключ формируется следующим образом:

rate_limit:login:{client_id}:{window}

где:

- client_id — идентификатор клиента (IP или фиксированное значение local для тестирования);
- window — номер временного окна (timestamp / 60).

**Значение ключа** — количество выполненных запросов в текущем окне.

Также для ключа устанавливается TTL ≈ 60 секунд, что автоматически сбрасывает счётчик при переходе в новое окно.

### Ограничения

Установлены следующие лимиты:

- 10 запросов в минуту на одного клиента для endpoint /auth/login

### Логика обработки запроса

При каждом запросе выполняются следующие шаги:

- Формируется ключ rate limit в Redis.
- Получается текущее значение счётчика.
- Если счётчик ≥ лимита:

возвращается ошибка:
```http
HTTP 429 Too Many Requests
```
Если лимит не превышен:

- счётчик увеличивается на 1;
- значение сохраняется в Redis с TTL;
- запрос обрабатывается дальше (логин пользователя).

### HTTP-заголовки

В ответах добавлены стандартные заголовки rate limiting:

- X-RateLimit-Limit       — максимальное число запросов (10)
- X-RateLimit-Remaining   — оставшееся число запросов
- X-RateLimit-Reset       — время до сброса (в секундах)

Пример ответа при превышении лимита
```http
HTTP/1.1 429 Too Many Requests

{
  "error": "Too many login requests"
}
```
### Проверка работы

Работа rate limiting была проверена с помощью серии запросов:
```bash
for i in {1..11}; do
  echo "Request $i"
  curl -i -s -X POST http://localhost:8080/api/v1/auth/login \
    -H "Content-Type: application/json" \
    -d '{"login":"user1","password":"pass123"}' | grep -E "HTTP/1.1|X-RateLimit|Too many|token"
  echo
done
```
#### Результат:

- первые 10 запросов — успешные (HTTP 200);
- 11-й запрос — отклонён (HTTP 429).

Также корректность работы подтверждена через Redis:
```bash
MacBook-Air-Kirill-2:lms_service kizhuravlev$ docker exec -it lms-redis redis-cli KEYS 'rate_limit:*'
1) "rate_limit:login:local:29624338"
MacBook-Air-Kirill-2:lms_service kizhuravlev$ docker exec -it lms-redis redis-cli GET "rate_limit:login:local:29624338"
"10"
MacBook-Air-Kirill-2:lms_service kizhuravlev$ docker exec -it lms-redis redis-cli TTL "rate_limit:login:local:29624338"
(integer) 26
```
### Вывод

Реализованный механизм rate limiting:

- защищает endpoint аутентификации от злоупотреблений;
- снижает нагрузку на систему;
- предотвращает brute-force атаки;
- корректно масштабируется за счёт использования Redis.

## 6. Анализ производительности

### Влияние кеширования и rate limiting

В ходе работы были внедрены два механизма повышения производительности системы:

#### Кеширование (Redis)

Кеширование позволяет:

- снизить нагрузку на базу данных (PostgreSQL);
- уменьшить время отклика за счёт обращения к памяти вместо диска;
- повысить пропускную способность системы.

На практике это проявилось следующим образом:

- при увеличении числа одновременных запросов RPS вырос;
- время обработки запроса снизилось;
- нагрузка на БД стала менее критичной.

Особенно эффективно кеширование работает для часто запрашиваемых данных (например, получение пользователя по логину).

#### Rate limiting

Rate limiting:

- ограничивает количество запросов от одного клиента;
- предотвращает перегрузку системы;
- защищает от brute-force атак (особенно на /auth/login).

Влияние на производительность:

- стабилизирует нагрузку на сервис;
- предотвращает деградацию системы при всплесках трафика;
- снижает вероятность отказов (timeouts, 5xx ошибки).

### Метрики для мониторинга

Для оценки производительности системы используются следующие метрики:

Основные метрики:
- Response Time (Latency)
  - Время обработки одного запроса (мс)

- Throughput (RPS)
  - Количество запросов в секунду


### Оценка эффективности кеширования (Cache Hit Rate)

Эффективность кеширования определяется метрикой cache hit rate.

Формула:
```
Hit Rate = Hits / (Hits + Misses)
```
где:

- Hit — данные найдены в кеше;
- Miss — данные отсутствуют в кеше и запрашиваются из БД.

#### Способы измерения:

- Через Redis

использование команд:
```bash
docker exec -it lms-redis redis-cli INFO stats
```
анализ:
```
keyspace_hits
keyspace_misses
```
- На уровне приложения

логирование попаданий и промахов:
```c++
if (cache_hit) hits++;
else misses++;
```
- Через мониторинг (Prometheus / Grafana)

сбор метрик:
```
cache_hits_total
cache_misses_total
```
Интерпретация:
- больше 80% — эффективный кеш
- 50–80% — средняя эффективность
- меньше 50% — кеш используется неэффективно

## Запуск проекта

### 1. Перейти в каталог сервиса

```bash
cd lms_service
```
### 2. Поднять PostgreSQL
```bash
docker compose up -d postgres
```
### 3. Инициализировать PostgreSQL
```bash
docker exec -i lms_postgres psql -U postgres -d lms_db < sql/schema.sql
docker exec -i lms_postgres psql -U postgres -d lms_db < sql/data.sql
```

#### Проверка корректности заполнения PostgreSQL

Подключиться к БД внутри контейнера
```bash
docker exec -it lms_postgres psql -U postgres -d lms_db
```
Ты попадёшь в `lms_db=#`

Для проверки количества записей в таблицах выполнить запрос:

```sql
SELECT 'users' AS table_name, COUNT(*) FROM users
UNION ALL
SELECT 'courses', COUNT(*) FROM courses
UNION ALL
SELECT 'lessons', COUNT(*) FROM lessons
UNION ALL
SELECT 'course_enrollments', COUNT(*) FROM course_enrollments
UNION ALL
SELECT 'lesson_progress', COUNT(*) FROM lesson_progress;
```

Результат должен показать, что в каждой таблице содержится по 10 записей,
что соответствует требованиям задания.

### 4. Поднять MongoDB
```bash
docker compose up -d mongo
```

#### Проверка корректности заполнения MongoDB
Подключиться к MongoDB:
```bash
docker exec -it lms_mongo mongosh
```

Внутри mongosh выполнить:
```bash
use lms_mongo_db
show collections
db.courses.countDocuments()
db.courses.findOne()
db.courses.getIndexes()
```

Ожидается, что коллекция courses существует и содержит 10 документов.

### 5. Поднять Redis

docker compose up -d redis

docker exec -it lms_redis redis-cli ping

PONG
